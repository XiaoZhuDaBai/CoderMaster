## 题目生成要求

### 1. 基本要求
- **题目必须严谨有解**：确保题目描述清晰，逻辑完整，存在唯一或明确的解
- **测试用例必须正确**：所有测试用例的输入输出必须经过验证，确保答案正确
- **场景化描述**：题目描述可以使用以下角色进行场景化：
  - **小猪**：通常是主角，遇到问题需要解决
  - **大白**：经验丰富的导师或朋友，给出提示或帮助
  - **小白**：新手角色，需要学习或练习
  - **大猪**：权威角色，设定规则或给出挑战
- **题目描述质量**：
  - 描述要清晰易懂，避免歧义
  - 包含必要的背景信息和问题陈述
  - 使用准确的技术术语，但保持可读性
  - 避免过于复杂或冗长的描述
- **题目原创性**：尽量生成新颖的题目，避免直接复制经典题目，可以在经典题目基础上进行变型或创新

### 2. 根据参数生成

#### 标签要求（tagIds）
根据提供的标签ID列表，题目必须涉及对应的算法或数据结构：
- 数据结构标签：数组、字符串、链表、栈、队列、树、图、哈希表、堆、并查集、字典树、线段树、树状数组、前缀和、差分数组、平衡二叉树等
- 算法标签：动态规划、贪心、回溯、分治、排序、搜索、深度优先搜索、广度优先搜索、双指针、滑动窗口、位运算、数学、模拟、二分查找、拓扑排序、最短路径、最小生成树、字符串匹配、数论、组合数学、几何、博弈论、构造、交互等
- 题目必须**充分体现**这些标签对应的知识点，不能只是简单提及

#### 难度要求（difficulty）
- **0-简单**：
  - 适合初学者，思路直接，实现简单
  - 时间复杂度通常O(n)或O(n²)，空间复杂度O(1)或O(n)
  - 数据规模：n ≤ 1000，通常可以直接暴力或简单遍历
  - 典型场景：数组遍历、字符串处理、简单模拟
- **1-中等**：
  - 需要一定的算法思维，可能需要优化
  - 时间复杂度O(n log n)或需要巧思，空间复杂度O(n)
  - 数据规模：n ≤ 10^5，需要优化算法
  - 典型场景：需要排序、二分、双指针、滑动窗口等技巧
- **2-困难**：
  - 需要复杂的算法设计，可能需要多种技巧结合
  - 时间复杂度优化到O(n)或更优，可能需要动态规划、图算法等
  - 数据规模：n ≤ 10^6或更大，必须使用高效算法
  - 典型场景：复杂动态规划、图论算法、高级数据结构、数学推导

#### 场景/来源（source）
如果提供了source参数，可以在题目描述中体现相应的风格：
- "竞赛"：题目描述可以更加正式，强调算法技巧
- "面试"：题目描述可以贴近实际工作场景
- "练习"：题目描述可以更加友好，适合学习

#### 额外要求（additionalRequirements）
如果提供了additionalRequirements，必须严格遵守：
- 例如："需要用到滑动窗口" → 题目必须使用滑动窗口算法
- 例如："要求时间复杂度O(n)" → 题目必须存在O(n)的解法
- 例如："需要用到动态规划" → 题目必须使用动态规划解决

请严格按照以下JSON格式输出，所有字段都必须提供；字段需要能够直接映射到数据库的题目表 `question` 和测试用例表 `question_test_case`：


**字段详细说明（同时说明与题目表 / 测试用例表的映射）：**

- **title**：题目标题，要简洁明了，能概括题目核心，20字以内；**对应 `question.title`**
- **description**：题目描述，200-500字，包含问题背景、问题陈述、要求等，可以使用角色场景化；**对应 `question.description`**
- **inputDesc**：输入描述，详细说明，**对应 `question.input_desc`**：
  - 输入格式（多行、单行、多组等）
  - 数据范围（所有变量的取值范围）
  - 约束条件（特殊限制、保证条件等）
- **outputDesc**：输出描述，详细说明，**对应 `question.output_desc`**：
  - 输出格式（每行一个结果、单行多个结果等）
  - 输出要求（精度、格式等）
- **examples**：样例数组的JSON字符串（需要双重转义），至少2个样例，每个包含；**整体对应 `question.examples`（JSON 字段）**：
  - input：输入样例（字符串，多行用`\n`分隔）
  - output：输出样例（字符串，多行用`\n`分隔）
  - explanation：样例解释（必须提供，帮助理解）
- **testCases**：测试用例数组，用于写入 `question_test_case` 表：
  - caseIndex：用例索引，从1开始递增
  - input：完整输入（字符串，多行用`\n`分隔）
  - expectedOutput：期望输出（字符串，多行用`\n`分隔）
  - isPublic：是否公开（1=公开，0=隐藏）
  - timeLimit：该用例的时间限制（通常为null，使用全局限制）
  - memoryLimit：该用例的内存限制（通常为null，使用全局限制）
- **tagIds**：标签ID数组，与请求参数一致；将用于写入 `question_tag` 表中的 `tag_id`
- **tagNames**：标签名称数组，必须与tagIds一一对应；用于展示和检查，不直接入库
- **difficulty**：难度值，0=简单，1=中等，2=困难；入库时可映射为 `"简单" / "中等" / "困难"` 存入 `question.difficulty`
- **questionType**：题目类型，固定为0（编程题），对应 `question.question_type`
- **source**：来源，与请求参数一致，如果没有则为null；对应 `question.source` 或题目元信息
- **timeLimit**：时间限制（毫秒），如果请求参数有则使用，否则根据难度设置；对应 `question.time_limit`
- **memoryLimit**：内存限制（MB），如果请求参数有则使用，否则根据难度设置；对应 `question.memory_limit`
- **stackLimit**：栈限制（MB），默认128；对应 `question.stack_limit`

**重要提示：**
- examples字段是JSON字符串，需要双重转义，例如：`"[{\"input\": \"1\\n2\", \"output\": \"3\", \"explanation\": \"解释\"}]"`
- testCases中的input和expectedOutput中的换行符要用`\n`表示（单个反斜杠加n）
- 所有字符串字段中的特殊字符要正确转义
- 确保输出的JSON格式完全正确，可以被标准JSON解析器解析

### 4. 测试用例要求（仅生成示例用例）

  - 解释清晰，能让用户在阅读后快速理解题意与判题流程；

    - 如果题目要求输入是多行，testCase 的 input 也必须是多行

    - 每个测试用例的 input 必须是一个完整的、可运行的输入！

**【通用歧义消除规则】- 适用于所有题目**：

**(A) 关于"选择"类操作**：
- 如果题目允许"选择是否"、"选择做或不做"，必须明确说明：

  - "必须按顺序选择，一旦开始不能跳过"（明确不能跳过）
- 示例对比：
  - ❌ 歧义："每天都可以选择是否消费"

  - ❌ 歧义："可以选择多个选项"
  - ✅ 明确："从 N 个选项中选择任意数量（可以为 0）"

**(B) 关于"连续"类操作**：
- 如果涉及"连续几天"、"连续选择"，必须明确：
  - "必须连续，不能间断"
  - "可以间断，不需要连续"

**(C) 关于"边界"类操作**：
- "至少"、"至多"、"不超过"等表述必须精确
- 示例：
  - ❌ 歧义："选择尽可能多的天数"
  - ✅ 明确："选择的天数尽可能多，但总花费不能超过 budget"

**(D) 验证描述与约束的一致性**：
- 检查题目描述是否与 inputDesc/outputDesc 中的约束一致
- 检查样例是否符合描述的规则

  - 每行一个结果，用换行符`\n`分隔
  - 多个数据在同一行：用空格分隔，末尾不要有多余空格

- **数据范围说明**：

  - 说明特殊约束：无重复、有序、非负等

   - 若任何条目不满足，先在草稿中修正，再输出最终结果，避免交付后返工。
12. **【通用验证规则】expectedOutput 必须经过严格验证**

   生成 expectedOutput 后，必须执行以下验证步骤（适用于所有题目类型）：

   **(A) 枚举验证法** - 列出所有可能的解/路径，对比得出最优：
   - 步骤1：列出所有可能的解法/选择路径
   - 步骤2：计算每种解法的结果
   - 步骤3：确认 expectedOutput 对应的是哪种解法
   - 步骤4：如果有多种可能的解法，必须明确说明哪种是最优解

   **(B) 边界验证法** - 检查边界条件和特殊情况：

   - 只有一个元素/步骤时的结果？
   - 最大/最小输入时的结果？

   **(C) 交叉验证法** - 用不同思路验证同一结果：
   - 用公式验证 vs 用枚举验证
   - 用贪心验证 vs 用动态规划验证

   **【强制要求】**：
   - 在输出 JSON 之前，必须在草稿中完成上述验证
   - 验证过程不需要输出，但必须执行
   - 如果发现 expectedOutput 与验证不符，必须修正后再输出
   - **禁止输出任何未经验证的 expectedOutput**

13. **示例口径统一**：
   - `examples` 与 `testCases` 的 input/output/explanation 必须严格对应同一套推导过程，禁止输出"答案应该是？"这类不确定或自我怀疑的措辞；
